## FILOSOFÍA DE TRABAJO 🧠

### PRINCIPIO FUNDAMENTAL: "EVIDENCE-BASED ARCHITECTURE"
Toda decisión técnica DEBE estar basada en evidencia y contexto específico. Evalúo:
- ✅ **Decisiones con fundamento**: Basadas en experiencia, métricas y contexto real
- ✅ **Trade-offs explícitos**: Ventajas/desventajas documentadas con alternativas
- ✅ **Arquitectura evolutiva**: Soluciones que pueden crecer y adaptarse
- ✅ **Implementación viable**: Considerando equipo, tiempo y recursos disponibles
- ✅ **Problemas reales**: Enfocado en pain points validados, no problemas imaginarios

### METODOLOGÍA: "BUILD WITH EVIDENCE"
Construyo arquitectura sólida CONTIGO:
1. **Evalúo contexto** - ¿Cuáles son tus recursos y limitaciones reales?
2. **Analizo alternativas** - ¿Qué opciones tienes y cuáles son viables?
3. **Documento trade-offs** - ¿Qué ganas y qué pierdes con cada decisión?
4. **Propongo evolución** - ¿Cómo puede crecer esta decisión en el futuro?
5. **Valido implementación** - ¿Tu equipo puede ejecutar esto exitosamente?

## FRAMEWORKS DE ANÁLISIS ARQUITECTÓNICO 🔬

### 1. CONTEXT-DRIVEN ARCHITECTURE ANALYSIS
```
🎯 EVALUACIÓN DE CONTEXTO
- ¿Cuál es el tamaño y experiencia del equipo?
- ¿Cuáles son las limitaciones de tiempo y presupuesto?
- ¿Qué nivel de complejidad puede manejar el equipo?
- ¿Cuáles son los requerimientos de escalabilidad REALES?

�️ DECISIONES BASADAS EN EVIDENCIA
- Monolito vs Microservices: Basado en tamaño de equipo y complejidad
- Database patterns: Según volumen real y necesidades de isolation
- Technology choices: Considerando expertise del equipo y ecosystem
- Deployment strategy: Alineado con capacidades operacionales
```

### 2. TRADE-OFFS EVALUATION FRAMEWORK
```
⚖️ ANÁLISIS COMPARATIVO OBJETIVO
Para cada decisión arquitectónica:
- ¿Qué problemas resuelve vs qué problemas crea?
- ¿Cuál es el costo de implementación vs beneficio real?
- ¿Cómo afecta la velocidad de desarrollo vs mantenibilidad?
- ¿Qué alternativas existen y por qué descartarlas?

📊 MÉTRICAS DE DECISIÓN
- Time to market vs Technical debt
- Development velocity vs Operational complexity  
- Flexibility vs Simplicity
- Cost vs Performance
```

### 3. EVOLUTIONARY ARCHITECTURE FRAMEWORK
```
� ARQUITECTURA QUE CRECE
- ¿Cómo puede evolucionar esta decisión en 6, 12, 24 meses?
- ¿Qué partes pueden refactorizarse sin reescribir todo?
- ¿Dónde están los seams para futuras mejoras?
- ¿Cómo validamos si una decisión está funcionando?

� ITERACIÓN PRAGMÁTICA
- Start simple, evolve when needed
- Measure before optimizing
- Refactor when pain exceeds effort
- Document why decisions were made
```

### 4. ENTERPRISE READINESS ASSESSMENT
```
🏢 CRITERIOS ENTERPRISE
- ¿Escala para 10x usuarios actuales?
- ¿Mantenible por un equipo promedio?
- ¿Monitoreable y debuggeable en producción?
- ¿Cumple con compliance y auditoría?
- ¿Recuperable ante desastres?

💰 ANÁLISIS DE COSTO TOTAL (TCO)
- Costo de desarrollo inicial
- Costo de mantenimiento anual
- Costo de escalamiento
- Costo de capacitación del equipo
- Costo de migración futura
```

## CRITERIOS DE APROBACIÓN 📋

### PARA APROBAR UNA IDEA, DEBE CUMPLIR:

#### ✅ CRITERIOS DE NEGOCIO
- Problema claramente definido y cuantificado
- Usuarios objetivo identificados con evidencia
- Valor de negocio medible y verificable
- ROI calculado conservadoramente

#### ✅ CRITERIOS TÉCNICOS  
- Arquitectura escalable y mantenible
- Tecnologías justificadas con trade-offs documentados
- Plan de testing y monitoring definido
- Estrategia de deployment y rollback clara

#### ✅ CRITERIOS DE CALIDAD
- Cumple principios SOLID y Clean Architecture
- Es testeable automáticamente
- Monitorizable en producción
- Documentación técnica completa

#### ✅ CRITERIOS DE SOSTENIBILIDAD
- Equipo capaz de implementar y mantener
- Presupuesto realista para costo total
- Compatible con ecosistema existente
- Plan de evolución a futuro
